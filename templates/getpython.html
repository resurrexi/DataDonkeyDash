A request for buy-in

Current State

  • Siloâ€™ed business units contributing to a project, but with different priorities and ownership in different processes of the project workflow
    • Barriers with workflows, adding to administrative costs and downtime
  • Multiple 3rd party software used in the project workflow
    • Time consumption due to manually exporting and converting data to a format native to the application or recognized by the next entity in the project workflow
    • Majority of 3rd party software (e.g. SAS) are expensive
    • Creative potential limited by rules and limitations of software
  • Visualizations and presentations still done on MS Office
    • Static charts and graphs... boring!
    • Files have to be transferred via email

Future State

  • Streamlined processes where the majority of a project's processes is run with one platform
    • Automatically eliminates barriers
    • Ownership falls to less business units, reducing downtime
  • An open source development platform with little/no limitations for any project
    • Elminates 3rd party software and expensive licenses
    • Promotes creativity and innovation
    • Encourages business unit to "eat its own dog food"
  • Visualizations and presentations delivered through the web
    • Interactive charts and graphs
    • Instant updates without the need to re-email an updated file
  • Large scale automation
    • Ability to automate every process with little/no IT barriers
    • Once a lot of mundane processes are automated, focus can then be shifted to more exciting projects (e.g. machine learning, deep learning)

Python bridges the states

  • Versatile language for developing any project of any size
  • Open source, with many developers and community support (e.g. StackOverflow)
  • Includes a growing number of packages from the developer community for doing... anything
    • If there isn't a package available, you can make one
  • Can be used with many different technology stacks

POC

  • HD Engagement Report
    • Backend data fetching using an open source ODBC driver that Python utilizes
    • Data is manipulated and formatted by Python before being sent to the front end for visual representation
    • Data sent to the front end is processed by a JS charting library that delivers the interactive graphs
  • Geocoding demo
    • Showcases Python's capability to query an external API and then GET the response from the API service
    • Response data is then consumed by another API that produces the map
  • Outreach demo
    • Uses a telephony API for making outbound SMS/voice calls
    • Eliminates usage of 3rd party vendors for doing outreach calls
    • Telephony API can be expanded to IVR applications

Current/Future Barriers

  • Corporate firewall blocks communications with some API's (e.g. Outreach demo).
  • Slow IO bound tasks (e.g. reading a database) can hold up web requests.
  • External APIs will have to be understood in order to be fully utilized.
  • Not many people here know Python and there is a learning curve. I recommend the following resources for personal development:
    • Take an online MOOC (e.g. Coursera, Udemy, Udacity)
    • Packt Publishing
    • Read the docs/manuals
    • Utilize StackOverflow
    • Practice and apply
  • PC hardware is not optimal to run as a production server. I recommend:
    • Minimum PC Specs
      • At least 32GB RAM
      • Quad core processor
      • SSD hard drive(s)
      • May have to utilize multiple servers to split Web/DB/Caching
    • Private cloud servers (e.g. AWS, IBM Bluemix)
